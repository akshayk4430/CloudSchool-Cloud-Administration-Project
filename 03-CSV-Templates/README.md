
# 📄 CSV Templates – CloudSchool Data Model

## 📌 Overview

This folder contains all CSV files used as input for user provisioning and identity management.

The design follows a structured approach where:

* CSV files act as the **source of truth**
* Identity data and business logic are **separated**
* Final provisioning uses a **merged dataset**

---

## 🧱 Data Model Design

The provisioning system is built using three layers:

### 1. Identity Layer

File:

```
staff.csv
```

Contains core identity information:

* UserPrincipalName
* DisplayName
* GivenName
* Surname
* MailNickname
* Department
* EmployeeType
* UsageLocation

This file represents the **base user identity data**.

---

### 2. Attribute Mapping Layer

File:

```
staff-attributes.csv
```

Contains business logic attributes:

* ExtensionAttribute1 → Role
* ExtensionAttribute2 → Assignment

Examples:

* Role: Principal, VicePrincipal, DeptHead, ClassTeacher
* Assignment: IT, Accounts, Operations, Grade1A

This layer allows flexible role and assignment management without modifying identity data.

---

### 3. Final Provisioning Layer

File:

```
staff-final.csv
```

Generated by merging:

* `staff.csv`
* `staff-attributes.csv`

This file is used by:

```
03-create-staff.ps1
```

It acts as the **single source of truth for provisioning**.

---

## 🎓 Student Data

Files:

```
students.csv
students-template.csv
```

These contain:

* UserPrincipalName
* DisplayName
* GivenName
* Surname
* Department
* EmployeeType
* UsageLocation
* ExtensionAttribute1 (Grade)
* ExtensionAttribute2 (Division)

Student data is already structured and does not require a merge step.

---

## 📊 Additional Files

### Raw Export (Intermediate)

```
staff-only-export.csv
```

* Original export from Entra ID
* Used for cleanup and transformation
* Not used directly for provisioning

---

## ⚙️ Workflow Summary

1. Export staff data → `staff-only-export.csv`
2. Clean and normalize → `staff.csv`
3. Define roles and assignments → `staff-attributes.csv`
4. Merge datasets → `staff-final.csv`
5. Provision users using final dataset

---

## 📌 Key Principles

* CSV files are the **source of truth**
* Identity data is separated from business logic
* Final dataset is **fully normalized before provisioning**
* Scripts are designed to be **idempotent**
* Changes should be made in CSV, not directly in scripts

---

## ⚠️ Notes

* Always validate CSV data before running scripts
* Ensure no duplicate UserPrincipalNames
* Maintain consistent naming conventions
* Test with a small dataset before full execution
